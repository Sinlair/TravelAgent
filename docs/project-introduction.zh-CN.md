## 项目定位

TravelAgent 是一个面向旅行规划场景的全栈工作台。
它的目标不是只返回一段聊天答案，而是把用户的旅行想法逐步转成可执行、可调整、可追踪的旅行方案。

这个项目适合下面几类需求：

- 想把自然语言旅行需求转成结构化行程
- 想把地图、天气、POI、知识库信息一起接入旅行规划
- 想把“生成方案”升级成“可编辑、可反馈、可回放评估”的产品工作台
- 想研究 DDD、多智能体编排、RAG、图片辅助输入在真实业务中的组合方式

一句话概括：
TravelAgent 是一个把“聊天式旅行需求”变成“结构化旅行方案”的多智能体旅行规划系统。

## 这个项目解决什么问题

很多旅行助手只能做一次性回答，存在几个常见问题：

- 只会聊天，不会给出结构化、可执行的 itinerary
- 生成结果缺少地图、天气、路线等真实世界信息支撑
- 用户修改需求后只能整段重来，缺少局部调整能力
- 前端只是消息流，没有行程、反馈、版本、质量运营这些产品能力
- 系统很难解释自己“为什么这么规划”以及“当前结果是否可靠”

TravelAgent 试图解决的正是这些问题：

- 把用户需求沉淀成结构化 trip brief
- 通过显式规划流水线生成旅行方案
- 用 Amap、高德相关能力做现实世界增强
- 用统一结果契约把聊天、行程、地图、反馈、时间线串成一个工作台
- 通过质量回放和反馈面板，把系统从 demo 推进到可持续迭代的产品形态

## 项目用了哪些技能/能力

这里的“技能”更适合理解为系统能力，而不是某个单独模型 prompt 技巧。

### 1. 多智能体路由能力

系统会先判断当前请求应该由哪个 specialist 处理，而不是把所有问题都交给一个通用 agent。

当前主要有四类 agent：

- `WEATHER`：天气与天气建议
- `GEO`：地理解析、地理编码、坐标问题
- `TRAVEL_PLANNER`：旅行规划、路线设计、预算、行程生成
- `GENERAL`：普通旅行相关问答

这样做的价值是：

- 路由更明确
- 每类问题的处理逻辑更稳定
- 后续更容易扩展新的 specialist

### 2. 结构化旅行规划能力

系统不仅返回文本，还会产出结构化 `TravelPlan`。

典型内容包括：

- 每日行程安排
- 景点与停留时段
- 酒店建议
- 预算区间
- 天气快照
- 知识库补充信息
- 交通与路线相关信息

这使它更像一个真正的旅行规划工具，而不是普通聊天机器人。

### 3. 真实旅行约束提取能力

系统会尝试把用户输入中的关键旅行约束提取出来并持久化，例如：

- 出发地
- 目的地
- 开始日期与结束日期
- 天数
- 预算
- 同行人
- 偏好

它还支持通过结构化 brief 直接修改这些字段，而不是只能靠自然语言反复“纠正”模型。

### 4. 图片辅助输入能力

项目支持上传旅行截图，并从截图中提取旅行事实，再进入规划链路。

这类能力适合处理：

- 订单截图
- 酒店确认页
- 行程单截图
- 票务信息截图

图片输入并不是独立能力，而是被并入了主工作流，让多模态输入最终仍落在统一的 trip brief 和 planner 上。

### 5. 地图与现实世界增强能力

系统集成了 Amap / 高德相关能力，用来增强旅行规划的现实可执行性。

主要包括：

- 天气快照
- POI 匹配
- 地理编码与逆地理编码
- 酒店片区建议
- 路线与交通信息

这一步让方案不只是“语言上合理”，而是更接近真实世界的旅行场景。

### 6. RAG 知识检索能力

系统会从本地旅行知识库，或可选的向量检索能力中补充目的地知识。

它的用途包括：

- 目的地背景知识补充
- 景点/区域相关建议
- 主题偏好匹配
- 让规划结果不只依赖在线工具返回的即时信息

### 7. 规划校验与修复能力

TravelAgent 不是“生成完就结束”。
在 planner 路径里，它会对生成结果做校验，并在必要时自动修复。

常见校验点包括：

- 预算是否超限
- 时间安排是否过满
- 是否存在重复或不合理停靠点
- 行程整体是否更接近可执行方案

这让系统具备“先生成，再检查，再修正”的能力。

### 8. 局部重规划能力

用户不需要每次都重做整份方案。
系统支持局部重规划，例如：

- 只调整酒店区域
- 只重算某一天

这是从“单轮回答”走向“可编辑产品”的关键一步。

### 9. 产品运营与质量评估能力

项目已经不只是“能生成计划”，还具备了产品化迭代所需的运营能力：

- 行前 checklist
- 方案版本快照
- 最近版本 diff
- 反馈原因标签
- 反馈运营过滤视图
- 质量场景回放脚本

这说明它已经从纯技术 demo，向内部 beta 级产品形态推进。

## 这个项目用了哪些设计方法

### 1. DDD 分层设计

项目整体采用了 DDD 启发下的分层方式：

- `travel-agent-domain`：领域模型、实体、值对象、端口接口
- `travel-agent-app`：用例编排、API、工作流
- `travel-agent-infrastructure`：具体实现、适配器、检索、持久化、模型调用
- `web`：前端工作台

这样做的好处是：

- 业务规则不容易被控制器和基础设施细节污染
- 核心模型更容易沉淀
- 替换外部能力时影响范围更可控

### 2. Ports-and-Adapters

这个项目不是把外部依赖直接写进核心逻辑，而是通过接口和适配器隔离：

- 地图能力通过 gateway / tool adapter 接入
- 持久化通过 repository adapter 接入
- 检索与向量能力通过独立实现接入
- 模型与工具提供者可以按配置切换

这使系统具备较好的替换性和可测试性。

### 3. 编排式多智能体设计

项目采用的是 orchestrated multi-agent，而不是多个 agent 自由对话。

特点是：

- 由 `ConversationWorkflow` 统一编排
- 由 `AgentRouter` 决定路由
- specialist 负责本领域执行
- 结果收敛到统一响应契约

这比“任由多个 agent 自己聊”更容易做产品化和调试。

### 4. 显式规划流水线

planner 不是一个黑盒 prompt，而是显式步骤：

1. 构造草案
2. 地图与知识增强
3. 校验
4. 修复
5. 生成最终结构化结果

这是一种非常重要的设计方法：把复杂生成任务拆成可观测、可替换、可验证的步骤。

### 5. 契约优先设计

`Herness` 改造的核心之一，就是统一聊天执行结果与持久化详情视图的结果契约。

重点字段包括：

- `feedbackTarget`
- `issues`
- `missingInformation`
- `constraintSummary`
- `timeline[*].status`

这样前后端围绕同一套契约协作，而不是各自猜测数据结构。

### 6. 可观测性优先

项目不是只关心“能不能返回结果”，还关注“过程能不能被看见”。

体现方式包括：

- timeline 事件
- SSE 流式过程输出
- planner 阶段状态
- 反馈和版本追踪
- 质量回放脚本

这是一种偏工程化和产品化的设计思路。

### 7. 降级与回退设计

项目没有假设所有外部依赖一定可用。
它内置了多种降级路径，例如：

- 无 OpenAI 时可走 `local-demo`
- 无 Amap key 时可走 mock
- 不同 tool provider 可在 `LOCAL / MCP` 之间切换
- 向量存储不可用时可回退到本地知识集

这说明它在设计上考虑了本地开发、测试和线上依赖不稳定等现实问题。

## 模块分工

### `travel-agent-domain`

负责：

- 领域实体
- 值对象
- 领域接口
- 核心业务语义

### `travel-agent-app`

负责：

- 工作流编排
- 控制器与 API
- SSE 输出
- 应用层服务

### `travel-agent-infrastructure`

负责：

- 模型接入
- 检索
- 持久化
- 校验与修复
- 工具和地图适配

### `travel-agent-amap`

负责：

- Amap / 高德 HTTP 能力接入

### `travel-agent-amap-mcp-server`

负责：

- 将 Amap 相关能力作为 MCP 工具服务暴露

### `web`

负责：

- 聊天工作台
- 行程展示
- 版本和 checklist 面板
- 反馈运营面板

## 一次请求是怎么跑起来的

以“帮我规划 2 天杭州旅行，预算 900 元”为例：

1. 前端把请求发送到 `/api/conversations/chat`
2. `ConversationWorkflow` 重建上下文和任务记忆
3. `AgentRouter` 把请求路由到 `TRAVEL_PLANNER`
4. planner 构造草案
5. 用 Amap 和知识检索做增强
6. 对结果做校验
7. 必要时触发修复
8. 输出文本答案、结构化 `travelPlan`、timeline、issues 等统一结果
9. 前端把结果呈现在聊天、行程、地图、反馈、版本等面板里

如果用户之后要求“只改酒店区域”或“重算第 2 天”，系统会在保留已有上下文的前提下做局部调整。

## 当前边界和限制

这个项目已经具备很强的产品原型能力，但仍然有边界：

- 不是机票酒店预订交易系统
- 不是支付系统
- 不是 OTA 全闭环平台
- 最强的地图 grounding 仍然偏中国旅行场景
- 真实效果仍然受模型配置、地图配置和知识数据质量影响

所以更准确地说，它是一个“旅行规划工作台”，而不是“旅行交易平台”。
