# 04. 设计方法

## 这一章讲什么

这一章关注的不是“用了什么技术栈”，而是“系统是按什么方法设计出来的”。

同样是 Java、Spring Boot、Vue、LLM，不同项目会做出完全不同的结构。
TravelAgent 的特点在于，它明显采用了一套偏工程化、偏产品化的设计方法组合。

## 1. DDD 分层设计

### 方法说明

项目整体采用了 DDD 启发下的分层结构：

- `domain`
- `app`
- `infrastructure`
- `web`

### 为什么要这么做

旅行规划并不是一个单纯的 API 拼装问题。
它涉及：

- 任务记忆
- 行程实体
- 版本快照
- 反馈目标
- 规划状态

如果不做分层，这些业务概念很容易散落在 controller、service、repository、前端转换逻辑中。

### 在项目中的体现

- 领域层放核心实体和值对象
- 应用层放工作流编排
- 基础设施层放外部能力实现
- 前端层放工作台展示和交互

### 方法收益

- 业务模型更容易沉淀
- 边界更清楚
- 可替换性更强
- 后续加能力时不容易把系统改乱

## 2. Ports-and-Adapters

### 方法说明

项目明显采用了 ports-and-adapters 思想。
也就是说，核心业务不直接依赖具体外部服务，而是通过端口和适配器连接。

### 为什么要这么做

这个系统要接入的外部能力很多：

- OpenAI 兼容模型
- Amap / 高德
- SQLite
- 可选向量库
- MCP 工具服务

如果没有适配器隔离，业务逻辑会被外部 API 细节污染。

### 在项目中的体现

- 地图能力通过 gateway / tool adapter 接入
- 持久化通过 repository adapter 接入
- 工具提供者支持 `LOCAL / MCP`
- 记忆提供者支持 `AUTO / SQLITE / MILVUS`

### 方法收益

- 能切换外部能力
- 本地和线上环境更容易区分
- 测试时更容易 stub / mock

## 3. 编排式多智能体设计

### 方法说明

TravelAgent 不是让多个 agent 自由聊天，而是采用 orchestrated multi-agent。

### 为什么要这么做

自由式 agent 协作在 demo 里可能很炫，但在产品里通常带来这些问题：

- 行为不可预测
- 很难调试
- 很难控制输出契约
- 很难让前端稳定消费

编排式设计更偏工程系统。

### 在项目中的体现

- `ConversationWorkflow` 统一编排
- `AgentRouter` 负责路由
- specialist 负责执行
- 最终结果回收成统一响应

### 方法收益

- 结构清晰
- 易观察
- 易调试
- 适合产品化

## 4. 显式规划流水线

### 方法说明

planner 不是一个黑盒，而是一条显式流水线。

典型步骤包括：

1. 构造草案
2. 地图与知识增强
3. 校验
4. 修复
5. 输出最终结果

### 为什么要这么做

旅行规划是一个高约束任务。
如果完全交给单次生成，很容易失控。

显式流水线的核心思想是：

- 把复杂任务拆成步骤
- 每一步都可观察
- 每一步都可替换
- 每一步都可局部优化

### 方法收益

- 更容易逐步提升质量
- 更容易解释过程
- 更适合后续产品演进

## 5. 契约优先设计

### 方法说明

`Herness` 改造体现的是典型的契约优先思路。
系统先统一结果契约，再围绕契约推进前后端一致性。

### 为什么要这么做

AI 系统常见问题是后端结果不稳定，前端只能不断猜字段。
这会导致：

- 状态混乱
- 面板不一致
- 维护成本高

### 在项目中的体现

统一字段包括：

- `feedbackTarget`
- `issues`
- `missingInformation`
- `constraintSummary`
- timeline 状态

### 方法收益

- 前后端更容易协作
- 结果更稳定
- 更适合长期维护

## 6. 可观测性优先

### 方法说明

项目不仅关注最终结果，还关注过程能不能被看到。

### 为什么要这么做

AI 系统常见抱怨是：

- 为什么这么慢？
- 为什么失败？
- 为什么改成这样？
- 为什么这次和上次不同？

如果没有过程可视化，这些问题很难回答。

### 在项目中的体现

- timeline 事件
- SSE 流式输出
- stage/status 语义
- feedback 追踪
- version diff
- quality replay

### 方法收益

- 用户体验更可解释
- 开发调试更高效
- 产品运营更可追踪

## 7. 降级与回退设计

### 方法说明

系统没有假设所有依赖始终可用，而是内置了多种回退路径。

### 为什么要这么做

AI 产品在现实里经常面对：

- 模型配置缺失
- 第三方服务失败
- 本地开发环境不完整
- 网络波动
- API key 不可用

如果没有降级设计，系统要么不能跑，要么极难调试。

### 在项目中的体现

- 路由可在无模型时启发式回退
- 任务记忆提取可启发式回退
- 图片理解可 fallback
- 工具提供者可切换 `LOCAL / MCP`
- Amap 可 mock
- 本地可用 `local-demo` profile

### 方法收益

- 开发和测试更稳
- 本地起服更容易
- 系统韧性更强

## 8. 产品化优先于技术炫技

### 方法说明

项目明显不是为了炫一个 agent 架构图，而是为了把能力做成工作台。

### 为什么要这么做

只有“会生成”的系统并不一定有产品价值。
真正的产品还需要：

- 编辑
- 追踪
- 反馈
- 版本
- 运营
- 评估

### 在项目中的体现

- trip brief editor
- checklist
- version snapshot
- recent diff
- feedback operations panel
- quality scenario replay

### 方法收益

- 更接近真实产品
- 更利于团队协作
- 更利于长期迭代

## 9. 质量场景驱动

### 方法说明

项目已经开始从“代码验证”走向“场景验证”。

### 为什么要这么做

AI 产品的很多失败不是类型错误，而是场景失败：

- 结果能返回，但不好用
- 格式没错，但规划不稳
- 某些城市表现强，某些城市表现弱

### 在项目中的体现

- quality scenarios fixture
- replay runner
- 结果导出
- feedback operations filters

### 方法收益

- 能从真实场景而不是单个函数看质量
- 更适合内部 beta 阶段

## 10. 前后端共享结果心智模型

### 方法说明

项目在设计上并没有把前端当成“被动展示层”，而是让前后端围绕同一结果心智模型协作。

### 为什么要这么做

如果前端只拿到一段文本，它几乎做不了复杂工作台。
而 TravelAgent 的目标是工作台，而不是对话框。

### 在项目中的体现

- 统一结果契约
- 共享 view model 思想
- 面板状态统一
- 反馈目标统一

### 方法收益

- 页面之间不容易打架
- 产品认知一致
- 扩展更容易

## 总结

TravelAgent 采用的设计方法不是单一流派，而是一组组合拳：

- DDD 保边界
- ports-and-adapters 保替换性
- 编排式多智能体保可控性
- 显式流水线保可观察与可优化
- 契约优先保前后端一致性
- 降级设计保系统韧性
- 产品化优先保真实落地能力

所以这个项目最值得学的地方，不是“用了哪些框架”，而是“如何把复杂 AI 能力组织成一个可运行、可维护、可演进的系统”。
