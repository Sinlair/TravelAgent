# 01. 项目定位

## 项目是什么

TravelAgent 是一个面向旅行规划场景的全栈工作台。
它不是单纯的“聊天机器人”，也不是完整的在线旅行交易平台，而是一个把旅行需求转成结构化方案的智能规划系统。

它的核心目标是：

- 让用户可以通过自然语言描述旅行需求
- 把需求逐步沉淀成结构化 trip brief
- 基于地图、天气、知识库和规划逻辑生成方案
- 把结果放进一个可编辑、可反馈、可追踪的产品工作台中

因此，TravelAgent 更像“旅行规划操作系统”里的一个工作台，而不是一个只会回答问题的对话框。

## 为什么要做这个项目

很多旅行产品和旅行助手都有一个共同问题：
它们能回答，但不一定能规划；它们能规划，但不一定可执行；它们即使能执行，也未必可追踪、可修改、可运营。

常见问题包括：

- 用户输入自由文本后，系统只能返回一段建议，缺少结构化结果
- 结果缺少真实世界增强，例如天气、路线、POI、酒店片区
- 生成后无法继续演进，只能重来
- 前端只有聊天界面，没有版本、反馈、时间线、检查清单等产品能力
- 系统输出过程不可见，用户看不懂结果可靠不可靠
- 团队难以评估一次改动是否让规划质量退化

TravelAgent 的设计目标就是正面解决这些问题。

## 它解决的核心问题

### 1. 从自然语言到结构化方案

用户通常不会按数据库字段思考旅行问题。
他们会说：

- “帮我规划一下杭州周末游”
- “预算 2000，两个人，想轻松一点”
- “我已经订好酒店了，帮我把行程顺一下”

系统需要把这些自然表达转成结构化约束，再进一步变成可执行计划。

### 2. 从“生成答案”到“生成工作流结果”

大多数对话式系统把结果理解成一句回答。
TravelAgent 把结果理解成一个工作流产物，至少包括：

- 文本回答
- 结构化行程
- 当前任务记忆
- 时间线
- 风险或缺失信息
- 可反馈目标

这意味着它天然更偏产品，而不只是一个模型 Demo。

### 3. 从“看起来合理”到“更接近真实”

旅行规划不是纯文本问题，而是现实约束问题。
一个看起来很顺的方案，可能会在现实中失败：

- 路线不顺
- 预算超标
- 一天安排过满
- 酒店位置不合理
- 景点分布不适合当天路线

所以系统必须接入地图、天气和目的地知识，提升方案的现实可执行性。

### 4. 从“一次输出”到“可持续迭代”

真正的旅行规划往往不是一次定稿。
用户会反复修改：

- 改预算
- 改天数
- 改酒店区域
- 改某一天行程
- 根据截图、订单、反馈继续细化

因此系统必须支持持久化、局部重规划、版本差异、反馈闭环和质量回放。

## 目标用户

这个项目面向的用户并不只有最终旅行者。

### 1. 终端用户

终端用户希望：

- 用自然语言表达旅行想法
- 得到清晰可执行的方案
- 能直接修改和确认关键约束
- 知道当前结果缺了什么、风险在哪里

### 2. 产品团队

产品团队希望：

- 不只是“能生成”，还要“能解释”
- 能看到用户为什么接受或拒绝方案
- 能跟踪哪些场景最容易失败
- 能持续做内部 beta 迭代

### 3. 工程团队

工程团队希望：

- 架构边界清楚
- 外部依赖可替换
- 工作流可观察、可调试
- 有回归验证，不至于越改越乱

### 4. 架构和研究人员

如果你关心这些主题，这个项目也有参考价值：

- 多智能体系统如何做成工程化产品
- DDD 在 AI 产品中的落地方式
- RAG、地图能力、图片输入如何拼成一个主流程
- 如何把“模型能力”变成“产品能力”

## 它不是什么

为了理解边界，更重要的是知道它不是什么。

### 不是 OTA 交易平台

它不负责：

- 机票交易
- 酒店交易
- 支付和退款
- 订单履约闭环

它更偏“规划和决策支持”，不是“交易和结算”。

### 不是纯地图产品

它虽然用了 Amap / 高德能力，但地图只是增强层。
核心目标依然是“生成旅行方案”，不是“做一个地图应用”。

### 不是单一大模型聊天壳

它没有把所有问题塞进一个大 prompt 然后期待模型自动完成。
相反，它更强调：

- 路由
- 分层
- 规划流水线
- 校验与修复
- 持久化和反馈

### 不是只面向开发者的技术演示

很多 AI 仓库更像技术样例。
TravelAgent 明显带有产品工作台取向：

- 前端是完整 workspace
- 有结构化 plan panel
- 有 checklist
- 有版本 diff
- 有反馈运营视图
- 有质量回放脚本

这说明它的目标是向真实产品形态推进。

## 它和普通聊天式旅行助手的差异

### 普通聊天式助手

通常是：

- 输入一句话
- 输出一段文本
- 结束

优点是简单，缺点是：

- 不结构化
- 难修改
- 不容易和前端工作台结合
- 不容易做版本和反馈分析

### TravelAgent

通常是：

1. 接收自然语言或图片输入
2. 提取约束和上下文
3. 路由到合适能力
4. 显式生成和增强计划
5. 校验并修复
6. 输出统一结果契约
7. 进入版本、反馈、质量回放闭环

所以它更像一个“旅行规划系统”，而不是“旅行问答机器人”。

## 当前产品阶段

从仓库内容看，这个项目不是停留在最初 MVP。
它已经明显进入了从 MVP 向内部 beta 产品化推进的阶段。

表现包括：

- 支持真实 trip brief 字段
- 支持行前 checklist
- 支持 plan version snapshot
- 支持 recent diff
- 支持 feedback operations 过滤
- 支持 quality scenario replay
- 支持 local-demo 启动模式

这说明它已经开始关注“可持续迭代”和“运营可见性”，而不只是“第一次跑通”。

## 为什么值得读

如果你想看一个比较完整的 AI 应用项目，这个仓库值得读的地方在于：

- 它不是纯 prompt 工程
- 它不是单纯前端壳
- 它不是只做 RAG
- 它不是只做 agent

它把这些东西组合成了一个相对完整的旅行规划工作台。

所以从学习角度看，它更适合研究：

- AI 能力如何进入业务流程
- 如何让结果结构化
- 如何做多模块协作
- 如何做产品化和工程化落地
