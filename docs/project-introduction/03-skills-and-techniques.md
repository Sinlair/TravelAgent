# 03. 技能与实现技巧

## 这一章怎么理解“技能”

这里的“技能”不是指某个 Codex 技能文件，也不是单独某个 prompt 技巧。
更准确地说，它指的是 TravelAgent 在系统层面具备的能力，以及这些能力背后的实现技巧。

如果要用一句话定义：
这里讨论的是“系统会什么”，以及“它凭什么会这些”。

## 一、输入理解相关技能

## 1. 意图识别与请求路由

### 系统会什么

系统能判断用户当前请求属于哪类任务：

- 天气
- 地理解析
- 旅行规划
- 普通旅行问答

### 它怎么做到

它并不只依赖单一路由方式，而是带有明显的混合策略：

- 有模型可用时，用 LLM 做更灵活的判断
- 无模型时，回退到启发式路由
- 通过关键词、目的地线索、预算线索、天数线索等规则兜底

### 为什么这很重要

这意味着系统不会因为单个依赖缺失就完全失去工作能力。
它不是“有模型就跑、没模型就死”，而是保留了可降级能力。

## 2. 任务记忆提取

### 系统会什么

系统会从对话中提取旅行相关字段，形成任务记忆。

### 它怎么做到

这里同样是双路径：

- 模型可用时，用结构化提取
- 模型不可用时，用启发式抽取

启发式抽取会识别：

- `from ... to ...`
- 中文目的地表达
- 天数字段
- 预算字段
- 偏好线索

### 技巧点

这个设计的关键不是“提取得多聪明”，而是“提取链路尽可能稳”。
对于工程系统来说，稳定比偶尔聪明更重要。

## 3. 图片事实提取

### 系统会什么

当用户上传截图时，系统会尝试抽取旅行事实。

### 它怎么做到

如果多模态模型可用，会走图片理解链路。
如果不可用，则回退到一个稳定的 fallback interpretation。

### 技巧点

它没有让图片能力脱离主流程，而是把图片事实提取结果统一投递回：

- 任务记忆
- 确认/忽略工作流
- 后续规划链路

这是一种“把多模态变成主流程输入”的技巧，而不是单做一个图片问答 Demo。

## 二、规划相关技能

## 4. 草案规划

### 系统会什么

系统能从任务约束构造初始旅行草案。

### 它怎么做到

planner 并不是一次性产出最终结果，而是先产出 draft，再进入增强、校验和修复。

### 技巧点

这是一种显式 pipeline 技巧：

- 先有草案
- 再有增强
- 再有校验
- 再有修复

这比“让模型一步到位”更容易观察和演进。

## 5. 目的地知识增强

### 系统会什么

系统能为当前目的地补充旅行知识。

### 它怎么做到

项目在知识检索方面使用的是分层和组合式技巧，而不是单一向量检索。

已有文档显示其核心思路包括：

- 查询理解
- 元数据增强
- 智能分块
- 混合检索
- 重排序

### 技巧点

这类实现不是“只做一个向量库搜索”，而是把 RAG 拆成多个质量控制环节。
它更像一个“检索流程”，而不是“检索函数”。

## 6. 地图 grounding

### 系统会什么

系统能结合地图服务增强旅行计划。

### 它怎么做到

主要技巧有：

- 把地图能力放在 planner 之后做 enrichment
- 支持 `LOCAL` 与 `MCP` 两种工具提供路径
- 对缺失 API key 或异常场景支持 mock/fallback

### 技巧点

这意味着地图能力不是强耦合死在一条路径里，而是：

- 可切换
- 可回退
- 可局部替换

这是一种典型的工程化技能。

## 7. 校验与修复

### 系统会什么

系统不只会生成，还会判断结果是否应该被接受，以及是否需要修复。

### 它怎么做到

planner 执行后会进入：

- `validate`
- `repair`
- 再验证

并且区分：

- 严格修复
- 放宽约束后的替代方案

### 技巧点

这是一种把生成任务做成“控制回路”的设计方式。
它不是一次生成，而是多轮内部自校正。

## 三、检索与排序相关技巧

## 8. 混合检索

### 系统会什么

系统既能利用关键词命中，也能利用语义相似度。

### 它怎么做到

根据现有文档，项目采用了：

- BM25
- 向量检索
- 加权融合
- rerank

### 技巧点

单一检索方式往往有偏差：

- 纯关键词容易漏语义
- 纯向量容易丢精确命中

混合检索是一种“取长补短”的技巧。

## 9. 查询理解

### 系统会什么

系统会对查询做理解和改写，以提高后续检索质量。

### 它怎么做到

相关模块会做：

- 查询改写
- 偏好理解
- 对话上下文引入

### 技巧点

RAG 质量的关键并不只是库里有什么，也在于“拿什么去搜”。
查询理解正是在提升这个入口质量。

## 四、产品工作台相关技能

## 10. 统一结果契约

### 系统会什么

无论后端内部怎么执行，前端拿到的是统一结果结构。

### 它怎么做到

通过 `Herness` 改造，系统标准化了：

- `feedbackTarget`
- `issues`
- `missingInformation`
- `constraintSummary`
- timeline 状态字段

### 技巧点

这是典型的契约优先技能：
先稳定结果结构，再让前后端围绕它协作。

## 11. 共享结果视图

### 系统会什么

聊天、行程、地图、反馈、版本等多个前端面板能围绕同一份结果视图协同工作。

### 它怎么做到

项目不是每个面板自己去拼装后端字段，而是引入共享 view model 思想。

### 技巧点

这是一种典型的前端产品化技巧：

- 降低重复逻辑
- 避免状态不一致
- 降低页面之间的认知冲突

## 12. 版本与反馈追踪

### 系统会什么

系统能把反馈绑定到稳定目标上，并保留计划版本。

### 它怎么做到

通过：

- plan version snapshot
- feedbackTarget
- target scope
- reason labels

### 技巧点

这让反馈从“用户点了一下按钮”升级成“可以回看、可以统计、可以运营”的数据。

## 五、工程与运维相关技巧

## 13. 配置驱动的能力切换

### 系统会什么

同一套系统可以根据配置切换不同运行路径。

典型开关包括：

- `travel.agent.tool-provider`
- `travel.agent.memory-provider`
- `travel.agent.knowledge.vector.enabled`

### 技巧点

这是一种非常实用的工程技巧：

- 本地开发可跑
- 测试环境可控
- 线上能力可切换
- 外部依赖可渐进引入

## 14. 本地离线 demo 启动能力

### 系统会什么

仓库支持 `local-demo` profile，在没有 OpenAI、MCP、Milvus 的情况下也能启动。

### 它怎么做到

通过：

- stub chat 依赖
- stub vector 依赖
- 本地工具路径
- mock Amap 数据

### 技巧点

这并不是“假装系统能跑”，而是把本地开发和链路验证正式纳入了工程设计。

## 15. 质量回放能力

### 系统会什么

项目支持预定义场景回放，并输出 JSON 与 Markdown 报告。

### 它怎么做到

通过：

- `quality-scenarios.json`
- `run_quality_scenarios.py`
- 可过滤的反馈和质量视图

### 技巧点

这使系统开始具备“场景级质量评估”能力，而不仅是“代码级单元测试”。

## 总结

TravelAgent 真正厉害的地方，不是它单独用了某一项技术，而是它把多种能力组合成了一个完整系统：

- 输入理解
- 规划生成
- 地图增强
- 检索补充
- 校验修复
- 工作台展示
- 反馈追踪
- 质量回放

如果只看一个点，它并不稀奇。
但把这些能力组合成一个相对完整、可运行、可演进的工作台，这才是它真正的“技能”。
