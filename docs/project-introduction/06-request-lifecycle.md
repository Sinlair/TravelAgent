# 06. 一次请求的生命周期

## 为什么这一章重要

理解一个 AI 产品，最有效的方法之一不是先看模块图，而是看“一次请求从输入到输出到底经历了什么”。

这一章就按真实工作流讲清楚：

- 普通文本规划请求怎么走
- 图片辅助请求怎么走
- 局部重规划怎么走
- 反馈怎么走
- SSE 时间线和持久化怎么参与

## 一、普通文本规划请求

假设用户输入：

“帮我规划一个 2 天杭州旅行，预算 900 元，想去西湖和吃本地美食。”

### 第 1 步：前端发起请求

前端工作台把请求发送到：

- `/api/conversations/chat`

请求体里可能包括：

- `message`
- `conversationId`
- `brief`
- `replanScope`

### 第 2 步：ConversationWorkflow 接管

`ConversationWorkflow` 是主入口。
它会先做一组标准化处理：

- 规范化消息文本
- 解析附件
- 处理结构化 brief patch
- 判断是不是局部重规划

这里的目标是把输入先整理成系统能稳定消费的内部形式。

### 第 3 步：重建会话和任务记忆

系统会根据 `conversationId` 找回当前会话状态，并重建任务记忆。

任务记忆里可能包括：

- 出发地
- 目的地
- 日期
- 天数
- 预算
- 偏好
- 同行人

如果用户这次请求带了新的结构化 brief，系统会把它合并进去。

### 第 4 步：建立路由上下文

系统会把下面几类信息组装起来：

- 当前用户输入
- 历史消息
- 任务记忆
- 会话摘要

这一步是后续路由和 specialist 执行的基础。

### 第 5 步：选择 specialist

`AgentRouter` 会判断当前请求属于哪一类。

对于旅行规划请求，通常会路由到：

- `TRAVEL_PLANNER`

如果模型不可用，路由器会回退到启发式逻辑，不至于整条链路失效。

### 第 6 步：planner 构造草案

进入 `TRAVEL_PLANNER` 后，系统先生成草案，而不是一步到位直接输出最终结果。

这一步的目标是：

- 先有一个可操作的 draft
- 后续再逐步加强质量

### 第 7 步：地图与知识增强

草案生成后，系统会做增强。

增强来源包括：

- Amap / 高德能力
- 目的地知识检索
- 天气快照

这一步会给草案补上更多现实世界语义。

### 第 8 步：校验

增强之后，系统会检查草案是否存在明显问题。

关注内容通常包括：

- 预算
- 行程密度
- 重复性
- 可执行性

### 第 9 步：修复

如果校验发现问题，系统不会直接结束，而是进入修复阶段。

可能发生两种情况：

- 严格修复后通过
- 放宽约束后形成替代方案

### 第 10 步：组装统一结果

当 planner 完成后，系统会把结果收敛成统一响应结构。

其中可能包括：

- `answer`
- `travelPlan`
- `taskMemory`
- `timeline`
- `issues`
- `missingInformation`
- `constraintSummary`
- `feedbackTarget`

### 第 11 步：持久化

系统会保存：

- 用户消息
- 助手消息
- 任务记忆
- 计划结果
- 时间线
- 版本快照

这样刷新页面后仍然可以恢复状态。

### 第 12 步：返回前端并渲染

前端拿到统一结果后，会把它拆到多个面板中呈现：

- 聊天结果区
- 行程面板
- 地图相关区域
- 时间线
- 反馈区
- 版本与 checklist 相关区域

## 二、图片辅助请求

图片流程比普通文本流程多了一个“待确认上下文”阶段。

## 1. 用户上传图片

当用户上传旅行截图时，系统不会直接把截图事实强行并入规划。

它会先做：

- 图片解释
- 生成摘要
- 生成结构化图片事实
- 把状态设为 pending

## 2. 进入待确认状态

系统会向用户返回：

- 已提取的图片事实摘要
- 待确认状态

目的是让用户先确认“这些信息是否真的应该进入规划”。

## 3. 用户确认或忽略

然后用户可以：

- 确认使用这些图片事实
- 忽略这批提取结果

如果确认，系统会把图片事实并入主工作流。
如果忽略，系统会丢弃这一批待确认上下文。

## 4. 再进入主规划流程

一旦确认，图片信息会像普通任务记忆一样进入 planner 流程。

这说明图片能力不是旁路，而是主流程的一部分。

## 三、局部重规划请求

局部重规划是这个项目区别于简单聊天机器人非常明显的一点。

## 1. 用户指定局部调整目标

用户可能发出类似请求：

- “酒店区域换一个”
- “只改第 2 天”

系统会把这类请求解析为 `replanScope`。

## 2. 读取现有计划

系统不会从零开始，而是先读出当前会话中已有计划。

## 3. 保留不受影响部分

局部重规划的关键不是“重新生成”，而是：

- 识别影响范围
- 保留未受影响部分
- 只重算目标片段

## 4. 生成新版本

局部调整完成后，系统会保存新的 plan version snapshot，并在前端提供最近 diff。

这让用户知道：

- 哪一部分被改了
- 哪一部分没动

## 四、反馈请求

反馈不是附属功能，而是完整生命周期的一部分。

## 1. 前端提交反馈

用户可以提交：

- 接受
- 部分接受
- 拒绝

并附带：

- 目标范围
- reason labels
- note

## 2. 后端绑定稳定目标

系统会把反馈绑定到稳定目标上，例如：

- answer target
- plan version
- scope

## 3. 反馈进入后续分析链路

这些反馈不仅用于“记录一次操作”，还会进入：

- 反馈汇总
- 运营过滤视图
- 质量分析

## 五、SSE 时间线如何参与

时间线不是事后生成的，而是在流程执行过程中逐步产生。

## 1. 关键阶段发布事件

在下面这些阶段，系统会发布 timeline 事件：

- 路由
- 调工具
- 校验
- 修复
- 完成

## 2. 前端实时接收

这些事件通过 SSE 推送到前端。

## 3. 同时被持久化

时间线事件不仅实时展示，也会被持久化。

这意味着：

- 刷新后还能看历史
- 可以对过程做回放和审计

## 六、失败和降级是怎么处理的

真实系统不可能总是一路成功。
TravelAgent 的生命周期设计里也考虑了失败和降级。

## 1. 路由失败

如果模型路由不可用，回退到启发式路由。

## 2. 图片理解失败

如果图片无法可靠解释，返回 fallback interpretation，而不是让流程崩掉。

## 3. 地图依赖不可用

如果地图 key 缺失或服务异常，可走 mock/fallback。

## 4. 规划无法完全满足约束

系统会明确区分：

- 已通过
- 修复后通过
- 仍存在风险

也就是说，它尽量不让失败变成“沉默失败”。

## 七、为什么这条生命周期有产品价值

这条生命周期真正的价值，不在于步骤多，而在于每一步都能服务产品目标：

- 输入可理解
- 中间过程可观察
- 结果可结构化
- 修改可局部化
- 反馈可追踪
- 质量可回放

这让系统从“生成一段内容”变成“支持一个产品工作流”。
