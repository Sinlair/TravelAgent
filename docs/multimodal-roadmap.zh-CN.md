# 多模态路线图

当前项目已经支持“文本 + 图片附件”输入。

但 planner 内部依然是文本优先：上传的图片会先被提取成旅行相关的文本上下文，再接入现有的路由、记忆、校验、修复和行程生成链路。

但这不代表多模态不适合这个项目。对行程规划助手来说，多模态真正的价值在于降低用户输入成本，把零散的旅行材料转成结构化规划上下文。

## 为什么值得做

高价值输入场景：

- 酒店预订截图
- 机票或火车票截图
- 景点海报或活动通知
- 地图截图
- 社交媒体攻略截图

这些输入可以帮助系统提取：

- 出发地和目的地
- 日期和时间
- 酒店名称和住宿区域
- 已预订活动
- 应该纳入或避开的 POI

## 建议范围

第一版只做图片输入。

不建议一开始就做：

- 语音聊天
- 视频理解
- 泛化的全媒体聊天

这些模式成本更高，而且对行程规划的核心价值不如图片输入直接。

## 第一阶段现状

第一版有用的图片输入链路已经落下来了：

1. 前端聊天流程支持图片上传
2. 后端请求协议接受图片附件
3. 会话消息会保存附件元数据
4. 图片内容会被提取成旅行相关的文本摘要
5. 提取结果会并入现有 planner 上下文
6. 用户可以先确认或忽略解析出的图片事实，再继续规划

当前还缺的部分：

1. 从“摘要提取”进一步提升到更强的结构化字段提取
2. 更明确的附件保留与删除策略
3. 对“图片输入是否提升 planner 接受率”的正式评估

## 非目标

- 用端到端视觉 prompting 替代现有 planner
- 在图片输入还没证明价值之前就支持任意媒体类型
- 把多模态只做成一个 UI 展示功能，而没有持久化和可追踪能力

## 风险

- OCR 噪声会污染日期、价格或地点名
- 截图里可能包含用户隐私数据
- 如果没有结构化抽取，图片输入会很难调试
- 多模态容易带来“看起来更聪明”的错觉，但 planner 本身未必真的更可靠

## 护栏

- 提取出的日期、预算和 POI 需要再次校验
- 在写入 itinerary 之前，先把解析结果展示给用户确认
- 原始附件和归一化后的 planner 字段要分开保存
- 继续保留现有 validate / repair 流程，不要绕过它

## 建议成功标准

- 用户上传预订或行程截图后，系统能得到可用的规划上下文，减少手工重复输入
- 提取结果能提升 planner 接受率，同时不增加硬约束违规
- 附件处理是可选增强，文本路径依然保持一等支持

## 建议顺序

1. 先把文本规划和反馈分析做稳
2. 继续把图片提取提升成更强的结构化字段
3. 评估图片输入是否真的提升 planner 接受率
4. 最后再决定要不要扩展到更广义的多模态支持
