# PRD: Herness 系统改造

## 1. 介绍 / 概述

`Herness` 是针对当前 TravelAgent 的一次系统级改造项目，覆盖前端工作台、后端对话编排链路、结构化输出契约、反馈闭环和回归保护能力。

本次改造不是新产品立项，而是在保留现有产品定位的前提下，解决以下核心问题：

- 行程规划链路虽然已经具备显式的 draft / enrich / validate / repair 流程，但对前端暴露的数据契约仍不够统一，导致结果展示、反馈记录和状态解释存在分散逻辑。
- 前端工作台已经具备聊天、结构化行程、地图、反馈和导出能力，但跨面板状态一致性不够强，用户对“当前答案是否完整、是否已修复、哪里失败了”的感知不稳定。
- 反馈闭环已有基础能力，但尚未稳定绑定到具体方案版本和问题类型，难以支撑后续产品调优与离线分析。
- 当前仓库正在持续演进 RAG、图像输入理解和规划质量，缺少一套围绕核心场景的统一回归保护，系统级改造风险较高。

本次改造的目标优先级定义如下：

- `P0`: 提升规划质量与输出一致性
- `P1`: 提升用户交互可理解性与执行可观测性
- `P2`: 提升工程可维护性与回归可控性

## 2. 目标

- 建立统一的会话结果契约，使前端对不同 specialist 的渲染逻辑收敛到同一套视图模型。
- 将规划链路中的关键阶段状态显式化并透出到 SSE 与持久化记录中，让用户和开发者都能判断当前结果经过了哪些处理。
- 让前端核心面板对 `loading / partial / success / empty / error` 五类状态具备一致表现，减少“有答案但无解释”的体验断层。
- 将反馈记录稳定绑定到具体回答版本、规划版本和原因标签，支撑后续产品分析与质量迭代。
- 为核心对话与规划场景建立可自动执行的回归验证基线，降低系统级改造后的隐性退化风险。

## 3. 用户故事

### US-001: 统一会话结果契约
**Description:** 作为前端开发者，我希望所有对话结果都遵循统一的数据契约，这样我可以用一致的方式渲染答案、结构化行程、地图和反馈区。

**Acceptance Criteria:**
- [ ] `/api/conversations/chat` 返回统一结果对象，至少包含 `agentType`、`answer`、`travelPlan`、`timeline`、`feedbackTarget` 和 `issues` 等稳定字段或等价字段。
- [ ] `WEATHER`、`GEO`、`GENERAL`、`TRAVEL_PLANNER` 四类 specialist 的返回结构在字段语义上保持一致，未提供的数据使用显式空值或空集合，而不是缺失字段。
- [ ] 前端消费层不再依赖 agent-specific 的散落判空逻辑来决定是否展示关键面板。
- [ ] 为新契约补充后端接口测试或序列化测试。
- [ ] 相关后端编译与测试通过。

### US-002: 标准化规划阶段事件
**Description:** 作为用户，我希望看到系统在“草案生成、增强、校验、修复、完成”各阶段做了什么，这样我能判断结果是否可靠。

**Acceptance Criteria:**
- [ ] 规划链路中的关键阶段使用统一事件模型表示，至少包含 `stage`、`status`、`message`、`startedAt`、`endedAt`。
- [ ] 阶段事件同时支持持久化记录和 SSE 推送。
- [ ] 当某阶段失败或触发 repair 时，事件中可以明确区分 `failed`、`repaired`、`completed` 等状态。
- [ ] 非规划类 specialist 不会伪造规划阶段事件，但返回结构中仍保留统一的 timeline 容器。
- [ ] 相关后端编译与测试通过。

### US-003: 暴露缺失信息与约束摘要
**Description:** 作为旅行规划用户，我希望系统明确告诉我当前方案缺少哪些信息、修复了哪些约束问题，这样我知道结果能否直接采用。

**Acceptance Criteria:**
- [ ] 当用户输入缺少关键旅行事实时，响应中包含结构化的缺失信息列表，例如目的地、出行时间、预算或同行人信息。
- [ ] 当规划经过校验与修复时，响应中包含约束摘要，至少能区分“通过”、“已修复后通过”与“仍有风险”三种状态。
- [ ] 缺失信息与约束摘要既可在最终回答中呈现，也可在结构化数据中被前端直接消费。
- [ ] 同一响应内不会出现文本描述与结构化字段相互矛盾的情况。
- [ ] 相关后端编译与测试通过。

### US-004: 重构前端统一结果视图模型
**Description:** 作为前端开发者，我希望聊天区、右侧行程区、地图区和反馈区共享一套结果视图模型，这样可以减少重复逻辑和状态不一致。

**Acceptance Criteria:**
- [ ] 前端为会话结果建立统一 view model 或等价适配层，而不是在多个组件中重复拼装字段。
- [ ] 聊天结果卡片、结构化行程面板、地图面板和反馈区读取的是同一份标准化结果数据。
- [ ] 同一条回答中的目的地、日期范围、预算提示等关键信息在不同面板中保持一致。
- [ ] `npm run build` 与相关前端测试通过。
- [ ] Verify in browser using dev-browser skill

### US-005: 完整覆盖核心面板状态
**Description:** 作为终端用户，我希望在任何处理中间态或异常态下都能知道系统当前处于什么状态，而不是只看到空白面板。

**Acceptance Criteria:**
- [ ] 聊天结果区、结构化行程区、地图区、反馈区至少覆盖 `loading`、`partial`、`success`、`empty`、`error` 五类状态。
- [ ] 当回答文本已生成但 `travelPlan` 或地图信息缺失时，界面显示“部分完成”而不是整体报错。
- [ ] 当 SSE 中断或后端返回错误时，界面有明确错误提示和可恢复操作，不要求用户刷新整页才能继续。
- [ ] 状态文案与视觉层级在全站保持一致。
- [ ] `npm run build` 与相关前端测试通过。
- [ ] Verify in browser using dev-browser skill

### US-006: 反馈闭环绑定到方案版本
**Description:** 作为产品分析人员，我希望每一次 Accept / Partially Accept / Reject 都能绑定到明确的回答版本和原因，这样后续分析才有可追溯性。

**Acceptance Criteria:**
- [ ] 新增或完善反馈目标标识，至少能关联到 conversation、message 或 plan version 中的一个稳定主键。
- [ ] `Partially Accept` 和 `Reject` 支持结构化原因标签，且可附带自由文本补充。
- [ ] 反馈记录可以区分用户是在评价文本答案、结构化行程，还是整体方案。
- [ ] 反馈导出或汇总接口中可读取原因标签与版本信息。
- [ ] `npm run build`、相关前端测试和相关后端测试通过。
- [ ] Verify in browser using dev-browser skill

### US-007: 可视化执行时间线
**Description:** 作为用户，我希望在工作台中查看一次请求的执行时间线，这样我可以理解系统为什么慢、为什么修复、为什么失败。

**Acceptance Criteria:**
- [ ] 前端提供时间线区域或等价展示，用于显示关键阶段、状态、时间戳和摘要消息。
- [ ] repair、fallback、tool-provider 切换等重要动作在时间线上可识别，而不是仅存在于后端日志。
- [ ] 时间线展示不会遮挡核心聊天与行程信息，在桌面端与移动端都能正常阅读。
- [ ] `npm run build` 与相关前端测试通过。
- [ ] Verify in browser using dev-browser skill

### US-008: 建立系统级回归基线
**Description:** 作为维护者，我希望有一组覆盖核心旅行场景的回归基线，这样 Herness 改造完成后可以持续防止质量回退。

**Acceptance Criteria:**
- [ ] 至少覆盖 10 个核心场景的自动化回归样例，包含普通规划、天气问答、地理解析、图像确认回流、规划修复、反馈提交等代表性路径。
- [ ] 回归样例能校验统一响应契约、关键结构化字段、阶段事件和错误处理分支。
- [ ] 前端至少覆盖统一 view model 和关键状态渲染的测试。
- [ ] CI 或本地标准命令可执行这些回归验证。
- [ ] 相关前后端测试命令在文档中可被开发者直接找到。

### US-009: 加固模块边界与改造约束
**Description:** 作为架构维护者，我希望 Herness 改造在清晰边界内完成，这样可以在提升能力的同时避免引入新的层间耦合。

**Acceptance Criteria:**
- [ ] `travel-agent-app` 只负责用例编排、API 与流式输出，不直接承载基础设施细节。
- [ ] `travel-agent-domain` 继续保持端口、实体和值对象边界，不引入面向前端的展示字段。
- [ ] `travel-agent-infrastructure` 中的 provider、gateway、repair、retrieval 能通过明确适配层接入，而不是穿透到上层控制器。
- [ ] 前端展示字段的新增由 DTO / view model 显式承接，不直接暴露内部基础设施对象。
- [ ] 为新增边界约束补充文档说明或测试保护。

## 4. 功能需求

1. `FR-1`: 系统必须为所有会话响应提供统一的顶层结果结构，以降低前端分支判断复杂度。
2. `FR-2`: 系统必须显式区分文本回答、结构化行程、时间线、反馈目标和问题摘要等不同信息块。
3. `FR-3`: 当 specialist 不产出结构化行程时，系统必须返回稳定的空值语义，而不是不确定字段缺失。
4. `FR-4`: 规划链路必须输出标准化阶段事件，并支持通过 SSE 实时传递给前端。
5. `FR-5`: 系统必须能标记规划结果是“直接通过”、“修复后通过”还是“仍存在风险”。
6. `FR-6`: 系统必须在用户输入不足时返回结构化缺失信息提示，指导用户补充必要事实。
7. `FR-7`: 前端必须基于统一视图模型渲染聊天、行程、地图、反馈和时间线区域。
8. `FR-8`: 前端必须覆盖核心面板的五类状态：`loading`、`partial`、`success`、`empty`、`error`。
9. `FR-9`: 反馈提交必须绑定到稳定的回答或方案标识，并支持原因标签。
10. `FR-10`: 反馈导出与统计接口必须可读取反馈原因和关联版本信息。
11. `FR-11`: 时间线展示必须让用户识别关键阶段、失败点、修复动作和最终完成状态。
12. `FR-12`: 系统必须为核心旅行场景提供自动化回归验证，并纳入标准开发流程。
13. `FR-13`: Herness 改造不得破坏现有 `LOCAL / MCP` 工具提供者切换能力。
14. `FR-14`: Herness 改造不得破坏现有会话持久化、SSE 流式输出和反馈汇总能力。
15. `FR-15`: 所有新增展示字段都必须通过明确 DTO 或适配层输出，不允许基础设施内部对象直接泄漏到前端。

## 5. 非目标

- 不在本次改造中引入全新的独立移动端 App。
- 不在本次改造中替换核心地图供应商或强制迁移到新的 LLM 供应商。
- 不在本次改造中重写整个 DDD 分层结构，也不追求“推倒重来”的大重构。
- 不在本次改造中建设完整运营后台或复杂 BI 平台。
- 不在本次改造中做模型微调、训练平台或大规模离线数据平台建设。

## 6. 设计考虑

- 保持当前“一屏工作台”的产品方向，不把关键链路拆成多页面流程。
- 在界面上优先突出“结果是否完整、是否修复过、是否存在风险”，而不是增加更多装饰性模块。
- 时间线、反馈和结构化行程应互相印证，避免用户看到多个相互竞争的信息源。
- 对 `Partially Accept` 和 `Reject` 的交互，应尽量使用低负担的标签选择，而不是只依赖长文本输入。

## 7. 技术考虑

- 需要在 `travel-agent-app` 中统一 API 输出模型，并评估对现有控制器、SSE hub 和 DTO 的影响。
- 需要在 `travel-agent-infrastructure` 中梳理 planner 阶段事件、校验结果和 repair 结果的标准表达方式。
- 需要在 `web` 中建立统一的会话结果适配层，减少多组件直接耦合后端字段。
- 需要确保 SQLite 持久化模型和反馈导出脚本能够兼容新增字段，必要时补充迁移方案。
- 需要补充跨模块回归测试，至少覆盖 `travel-agent-app` 与 `web` 两侧的契约和渲染逻辑。
- 如果改造引入新字段或新状态，必须同步更新开发文档和架构说明，避免实现与文档再次脱节。

## 8. 成功指标

- 100% 新会话响应遵循统一顶层结果契约，不再依赖 agent-specific 缺省字段行为。
- 至少 10 个核心场景具备自动化回归验证，且在本地标准命令或 CI 中可执行。
- 前端 5 类关键状态在聊天、行程、地图、反馈区全部有明确表现，不出现“空白但无解释”的核心路径。
- 100% 新提交的反馈记录都能关联到稳定的回答或方案标识，并带有明确反馈类型。
- 对一次典型规划请求，用户能够在界面中看到从生成到增强、校验、修复、完成的完整过程或明确缺失原因。

## 9. 开放问题

- `Herness` 是否需要作为独立版本旗标逐步灰度，还是直接替换现有默认链路？
- 反馈原因标签需要采用固定枚举，还是允许项目初期先用“固定标签 + 自由文本”的混合方案？
- 时间线是否只展示用户可理解的摘要事件，还是要保留面向开发者的技术事件视图？
- 对旧历史会话，是否需要补齐统一结果契约字段，还是只保证新产生的会话记录兼容？
- 当前缺少基线业务指标，本次改造是否需要在实施前先补一次现状测量，以便后续验证质量提升幅度？
