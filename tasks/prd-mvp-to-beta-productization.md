# PRD: Travel Agent 产品化补强（从 MVP 到可内测版本）

## 1. 介绍 / 概述

当前 Travel Agent 已经具备聊天规划、截图输入、结构化行程、高德增强、时间线可视化、反馈闭环和手账导出能力，已经是一个功能完整的旅行规划 MVP。

但如果目标是从当前 MVP 进入“可内测版本”，它仍然缺少几类关键产品能力：

- 用户不应该依赖“会提问”才能获得高质量结果，系统需要显式采集真实出行约束，例如出发地、具体日期、同行人和预算边界。
- 生成结果虽然已有结构化行程和时间线，但对普通用户来说，仍然不够“可信”和“可执行”，缺少统一的证据、置信度、估算/回退说明和需要二次确认的关键点。
- 真正可用的旅行产品不止要生成方案，还要帮助用户进入“出发前准备”阶段，包括待确认事项、预订状态、票务提醒和行前检查。
- 产品团队需要比“能生成计划”更强的运营与质量闭环，能够识别哪些城市、哪些场景、哪些问题类型最容易失败，从而支持内测期快速迭代。

本次 PRD 只聚焦从 MVP 到可内测产品的关键补强，目标用户优先为中国自由行普通用户，范围覆盖用户侧产品功能和运营侧质量数据闭环。

## 2. 目标

- 让新用户无需掌握提示词技巧，也能在首轮或前两轮内补齐核心旅行约束并生成可执行草案。
- 让用户能快速判断方案中的哪些推荐是“高可信”、哪些是“估算”或“需要确认”的内容。
- 让生成结果从“好看的方案”升级为“可以开始执行的出发前工作台”。
- 为内测期建立可持续的质量观测能力，让团队能按城市、场景、失败类型和反馈原因持续优化。
- 为后续公开发布打下产品基础，但不在本阶段引入复杂交易和支付能力。

## 4. 功能需求

1. `FR-1`: 系统必须支持记录真实旅行约束，包括出发地、开始日期、结束日期、预算、同行人和偏好。
2. `FR-2`: 用户必须可以在 UI 中直接编辑核心旅行约束，而不只能依赖自然语言补充。
3. `FR-3`: 当存在真实日期时，行程必须显示按日期展开的每日安排。
4. `FR-4`: 当缺少真实日期时，系统必须明确标记当前方案是估算模式。
5. `FR-5`: 酒店、POI 和路线等关键推荐必须提供来源、可信度和 fallback 标识或等价字段。
6. `FR-6`: 前端必须统一展示推荐可信度，且低可信度内容必须有显式提醒。
7. `FR-7`: 系统必须支持局部修正，而不是所有修改都触发整份方案重做。
8. `FR-8`: 每次生成结构化行程时，系统必须同时提供行前准备清单。
9. `FR-9`: 行前准备清单必须支持状态切换，并在 UI 中反映剩余待办。
10. `FR-10`: 系统必须在方案重算后保存版本快照，并允许反馈绑定到具体版本。
11. `FR-11`: 前端必须支持查看最近版本之间的核心差异。
12. `FR-12`: 产品团队必须能维护标准场景基线，并用统一命令复跑。
13. `FR-13`: 反馈运营面板必须支持按城市、原因标签、agent 类型和版本进行筛选。
14. `FR-14`: 运营面板必须能导出结构化数据供离线分析。
15. `FR-15`: 新增产品化字段必须在聊天响应和会话详情响应中保持一致，避免前端刷新后状态丢失。

## 5. 非目标

- 本阶段不做真实机票 / 酒店预订交易闭环。
- 本阶段不做支付、退款、订单履约等电商能力。
- 本阶段不做用户社交内容社区或攻略内容平台。
- 本阶段不做复杂会员体系、积分体系或商业化广告投放。
- 本阶段不做跨境多地图供应商切换和全球化能力重构。

## 6. 设计考虑

- 面向中国自由行普通用户，交互应优先“直观、少学习成本”，不要假设用户会写复杂 prompt。
- 可信度与证据展示必须是可感知的，不应该埋在折叠细节里才看得到。
- 行前清单要强调“下一步做什么”，而不是再输出一块冗长信息。
- 版本对比应聚焦“用户关心的差异”，不要做开发者式原始 JSON diff。
- 运营面板面向产品和运营，而不是只给研发看日志。

## 7. 技术考虑

- 需要扩展任务记忆、会话详情和聊天结果契约，保持前后端字段一致。
- 需要为结构化推荐对象定义统一的来源 / 可信度 / fallback 表达方式。
- 需要评估版本快照的存储方式，避免直接覆盖唯一一份 `TravelPlan`。
- 需要在反馈模型中稳定关联版本、作用域和原因标签。
- 需要补充场景基线与自动化评测脚本，避免产品化改造导致规划质量回退。
- 需要保证新增产品化字段对旧数据兼容，尤其是本地 SQLite 持久化和历史会话读取。

## 8. 成功指标

- 80% 以上核心内测场景能在前两轮内补齐关键旅行约束并输出结构化草案。
- 100% 关键推荐对象都能展示来源或 fallback 标识。
- 100% 反馈记录都能关联到稳定版本和反馈范围。
- 运营团队能在 5 分钟内定位最近一周最常见的失败原因和城市分布。
- 在标准基线场景中，高风险约束问题比例较当前版本显著下降。

## 9. 开放问题

- 日期字段是先支持“开始日期 + 天数”还是直接支持完整日历范围？
- 同行人类型需要做固定枚举，还是允许自由文本 + 标签混合模式？
- 行前清单的“完成状态”是只存在本地当前会话，还是需要后端持久化？
- 版本对比初期只支持最近两个版本，还是需要支持任意版本选择？
- 运营面板是直接扩展当前反馈汇总页，还是单独做新的质量工作台？

## 10. 里程碑 / 版本拆分

### Phase P0：让方案更像真实出行方案

- 旅行约束字段扩展
- 旅行简报编辑区
- 日期化行程窗口
- 推荐可信度字段与前端展示

### Phase P1：让方案进入可执行准备

- 局部修正能力
- 行前准备清单生成
- 行前清单交互面板
- 方案版本快照与最近版本对比

### Phase P2：让产品具备可持续内测优化能力

- 标准质量场景基线
- 反馈汇总增强为质量运营面板
- 结构化导出与离线分析流程
